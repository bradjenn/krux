**Step 1: Create `backend/src/services/sessionRegistry.ts`.**

Implement `SessionRegistry` as a module-level singleton object backed by `new Map<string, SessionRecord>()`:

- `create(projectId: string): SessionRecord` — generate UUID via `randomUUID()`, set status to `'idle'`, set `createdAt` to `Date.now()`, add to Map, return the record.
- `get(id: string): SessionRecord | null` — return from Map or null.
- `update(id: string, patch: Partial<Omit<SessionRecord, 'id'>>): boolean` — spread patch onto existing record. Return false if id not found.
- `delete(id: string): boolean` — return `registry.delete(id)`.
- `list(): SessionRecord[]` — return `Array.from(registry.values())`.
- `listByProject(projectId: string): SessionRecord[]` — filter by projectId.
- `clear(): void` — clear the Map (useful for testing).

Import `SessionRecord` type from the types re-export: `import type { SessionRecord } from '../types/index.js'`.

**Step 2: Create `backend/src/lib/processLifecycle.ts`.**

PID file location: `backend/data/.gsd-pids` (same data dir as projects.json). Resolve using `import.meta.dirname` relative path to `../../data/.gsd-pids`.

Implement these exports:

`readPids(): number[]`
- Read PID file as JSON. Return `[]` on any error (ENOENT, parse error).

`writePids(pids: number[]): void`
- Write the array as JSON to the PID file. Use `mkdirSync` + `writeFileSync`. No need for atomic write here — PID file corruption just means cleanup is skipped (acceptable).

`async cleanupOrphans(): Promise<void>`
- If PID file doesn't exist, return immediately.
- Read PIDs. For each, check existence with `process.kill(pid, 0)` (wrapped in try/catch). If process exists, send `SIGTERM` (wrapped in try/catch — process may vanish between check and kill).
- Log each kill: `console.log(\`Killed orphaned Claude process (PID ${pid})\`)`.
- Always delete PID file after cleanup sweep (`fs.unlinkSync` in try/catch).
- If any were killed, log summary: `console.log(\`Orphan cleanup complete: killed ${killed} process(es)\`)`.

`setupSigtermHandler(server: Server): void`
- Accept `server` parameter typed as `import('node:http').Server`.
- Create a `shutdown(signal: string)` function:
  1. Log: `console.log(\`Received ${signal}, shutting down...\`)`.
  2. Read PIDs from file.
  3. Send SIGTERM to each PID (try/catch, ignore ESRCH).
  4. Set a 5-second timeout. When it fires: for each PID that still exists (`process.kill(pid, 0)`), send SIGKILL. Then `server.close(() => process.exit(0))`.
  5. Call `forceKillTimer.unref()` so the timer alone doesn't keep the process alive.
  6. Immediately call `server.close(() => { clearTimeout(forceKillTimer); process.exit(0) })` — if server closes before 5s, cancel the force kill timer and exit.
- Register: `process.on('SIGTERM', () => shutdown('SIGTERM'))` and `process.on('SIGINT', () => shutdown('SIGINT'))`.

Per CONTEXT.md locked decisions:
- Kill orphans at server startup ONLY (no periodic sweep).
- Use PID file tracking (`.gsd-pids`).
- Graceful shutdown: SIGTERM to agents, wait up to 5 seconds, then SIGKILL.
- Log cleanup actions to stdout.

**Step 1: Create `backend/src/routes/sessions.ts`.**

Create a Hono sub-app exported as `sessionsRoute`:

- `GET /` — call `SessionRegistry.list()`, return JSON array. Optionally filter by `?projectId=` query parameter using `SessionRegistry.listByProject()`.
- `POST /` — parse body `{ projectId }`, validate projectId is present (return 400 if not), verify project exists via `ProjectStore.get(projectId)` (return 404 if not), call `SessionRegistry.create(projectId)`, return 201 with session record.
- `GET /:id` — call `SessionRegistry.get(id)`, return 404 if null, else return session record.
- `DELETE /:id` — call `SessionRegistry.delete(id)`, return 204 on success, 404 if not found.

All error responses use the locked error envelope: `{ error: { code: string, message: string } }`.

Import `ProjectStore` from the project store service to validate projectId exists.

**Step 2: Update `backend/src/index.ts`.**

Add these changes to the server entry point created in Plan 01-01:

1. Import `cleanupOrphans` and `setupSigtermHandler` from `./lib/processLifecycle.js`.
2. Import `sessionsRoute` from `./routes/sessions.js`.
3. Before the `serve()` call, add `await cleanupOrphans()`. Since this is a top-level await in an ESM module, it runs before the server starts accepting requests.
4. Mount the sessions route: `app.route('/api/sessions', sessionsRoute)`.
5. Inside the `serve()` callback (the second argument), add `setupSigtermHandler(server)` AFTER the "running" log message. This ensures the handler is registered only after the server is confirmed listening.

The updated index.ts flow should be:
```
await cleanupOrphans()     // Step 1: kill orphans
const app = new Hono()     // Step 2: configure app
// ... middleware, routes ...
serve({ ... }, (info) => { // Step 3: start server
  console.log(...)
  setupSigtermHandler(server)  // Step 4: register shutdown
})
```

IMPORTANT: Register signal handlers inside the `serve()` callback, NOT before the `serve()` call. The serve callback fires after the server is confirmed listening. Registering before can cause "Server is not running" errors on shutdown (see pitfall 2 from research doc).
