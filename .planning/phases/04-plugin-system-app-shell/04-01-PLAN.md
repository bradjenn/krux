<objective>
Clean up the plugin API, remove the sidebar plugin zone, add tab auto-open behavior, and introduce a lifecycle event bus.

Purpose: The existing plugin system works but has unnecessary complexity (sidebar plugin zone, no availability gating). This plan simplifies the PluginDefinition type, removes GsdSidebar and the PluginSidebar component from the sidebar, adds disabled state to the tab dropdown for unavailable plugins, wires auto-open for GSD tabs on project select, and introduces a mitt event bus for plugin lifecycle events.

Output: A cleaner plugin API that matches the tab-provider model, a sidebar that is project-list-only, a tab dropdown with availability gating, auto-open behavior, and a typed event bus.
</objective>

**2. Update `src/components/layout/Shell.tsx` — auto-open GSD tab on project select:**

- Import `PLUGINS` from `@/plugins`
- Import `invoke` from `@tauri-apps/api/core`
- Add a new `useEffect` AFTER the existing auto-create-terminal effect. This effect handles plugin auto-open:
  ```typescript
  useEffect(() => {
    if (!activeProjectId) return
    const project = projects.find((p) => p.id === activeProjectId)
    if (!project) return

    // Check each plugin with autoOpen enabled
    PLUGINS.filter((p) => p.autoOpen).forEach(async (plugin) => {
      const available = plugin.isAvailable ? await plugin.isAvailable(project.path) : true
      if (!available) return

      const tabType = plugin.defaultTabType || plugin.tabTypes[0]?.id
      if (!tabType) return

      const { tabs, addTab, setActiveTab } = useAppStore.getState()
      const existing = tabs.find((t) => t.type === tabType && t.projectId === activeProjectId)
      if (existing) {
        // Focus existing tab instead of creating duplicate
        setActiveTab(existing.id)
        return
      }

      addTab({
        id: crypto.randomUUID(),
        type: tabType,
        label: plugin.name,
        projectId: activeProjectId,
      })
    })
  }, [activeProjectId])
  ```
- This effect runs on project select. It waits for the async `isAvailable` check, then either focuses the existing GSD tab or creates a new one.
- The Terminal auto-create effect still fires independently (they don't conflict — Terminal runs first synchronously, GSD runs async after availability check).
- The GSD tab should end up focused (since `addTab` sets `activeTabId`). If both Terminal and GSD are created, GSD tab will be the active one (last `addTab` wins), which is the desired behavior per CONTEXT.md: "GSD tab focuses if `.planning/` exists".
